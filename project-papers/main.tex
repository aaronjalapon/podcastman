\begin{abstract}
   Consuming written content while on the move is increasingly popular, yet many
   high-quality blog articles lack an audio counterpart.  This paper presents an
   end-to-end AI system that transforms any blog post into a two-host podcast
   dialogue and synthesises natural-sounding speech via Google Cloud
   Text-to-Speech.  The pipeline follows a four-layer architecture:
   (1) input \& pre-processing with custom semantic chunking, (2) retrieval
   via a ChromaDB vector store with Jina~AI embeddings, (3) generation with
   four specialised Retrieval-Augmented Generation agents orchestrated by a
   LangGraph state machine, and (4) two-voice audio output using
   Google Cloud TTS with Standard voices and SSML prosody control.  A
   Streamlit frontend and FastAPI backend provide a complete web interface for the
   blog-to-podcast workflow.
\end{abstract}

\section{Introduction}\label{sec:intro}
The explosion of online written content---blogs, tutorials, news articles---has created a demand for alternative consumption modalities.  Podcasts, audiobooks, and voice-assistants enable users to listen while commuting, exercising, or multitasking.  However, manually rewriting blog articles as audio scripts is labor-intensive and error-prone.  Large Language Models (LLMs) have shown remarkable ability to understand and produce natural language, yet naïve prompting often leads to hallucinations or overly generic output \cite{Lewis2020RAG}.  

Retrieval-Augmented Generation (RAG) mitigates these issues by grounding LLM responses in a searchable knowledge base \cite{Gao2022RAGSurvey}.  When combined with a \textbf{multi-agent orchestration} framework, RAG can decompose a complex transformation task into well-defined subtasks---extracting salient ideas, drafting a conversational script, and adding directorial cues---while keeping each step anchored to the original text.  

This work proposes a full-stack system that (i) ingests a blog URL, raw text, or markdown, (ii) builds a ChromaDB vector store of semantically meaningful chunks, (iii) runs four specialised RAG agents---\textit{Script Generator}, \textit{Accuracy Agent}, \textit{Storytelling Agent}, and \textit{Engagement Agent}---orchestrated by a LangGraph state machine to produce a polished two-host podcast dialogue, and (iv) synthesises expressive two-voice speech using Google Cloud Text-to-Speech \cite{GoogleCloudTTS2024} with Standard voices and SSML-driven prosody control.  The result is a one-click ``blog-to-podcast'' service, served by a FastAPI backend and a Streamlit frontend, that can be integrated into learning platforms, news aggregators, or personal knowledge-management tools.

\section{Objectives}
\label{sec:obj}
\subsection{General Objective}
\begin{quote}
   To develop an automated pipeline that converts any blog article into a high-quality, podcast-ready audio file by leveraging Retrieval-Augmented Generation and AI voice synthesis.
\end{quote}

\subsection{Specific Objectives}
\begin{enumerate}[nosep, leftmargin=*]
  \item Design a robust ingestion and chunking module that extracts clean, semantically coherent text segments from heterogeneous blog sources using \texttt{newspaper3k}, \texttt{httpx}, and a custom paragraph-boundary chunker (800-token chunks, 100-token overlap).
  \item Build a ChromaDB vector database powered by \textbf{Jina~AI} \texttt{jina-embeddings-v3} embeddings (via LiteLLM) to enable accurate content retrieval.
  \item Implement four RAG agents---\textit{Script Generator}, \textit{Accuracy Agent}, \textit{Storytelling Agent}, and \textit{Engagement Agent}---orchestrated by a LangGraph \texttt{StateGraph} to generate a coherent, engaging two-host podcast dialogue.
  \item Integrate Google Cloud Text-to-Speech with Standard voices (\texttt{en-US-Standard-J} and \texttt{en-US-Standard-H}) and SSML prosody control to synthesise natural, expressive two-voice audio (HOST\_A and HOST\_B) while preserving speaker consistency.
  \item Define evaluation criteria including (a) content fidelity (BLEU/ROUGE vs.\ human-crafted scripts), (b) audio quality (MOS), and (c) end-to-end latency for future quantitative assessment.
\end{enumerate}

\subsection{Data Ingestion \& Pre-processing}
\begin{itemize}
  \item \textbf{Input} -- Blog URL, raw text, or markdown.  URLs are fetched via \texttt{httpx} and parsed primarily with \texttt{newspaper3k} (\texttt{Article}), with \texttt{BeautifulSoup} as a fallback extractor.
  \item \textbf{Cleaning} -- Custom regex normalisation in 
  \texttt{parser.py} (\texttt{normalize\_text()}, \texttt{parse\_markdown()}) strips extraneous whitespace, HTML artefacts, and boilerplate.  Markdown input is also supported natively.  Sentence segmentation (\texttt{segment\_sentences()}) uses \textbf{NLTK} \texttt{sent\_tokenize()} as the primary splitter, with a regex fallback.
  \item \textbf{Chunking} -- A custom \texttt{chunk\_text()} function splits the cleaned text into overlapping chunks of 800 tokens (overlap 100) by splitting at paragraph boundaries first, falling back to sentence and word boundaries.  Token counting uses \texttt{tiktoken}.  Overlap is approximated at {\raise.17ex\hbox{$\scriptstyle\sim$}}1.3 tokens per word for efficiency.
\end{itemize}

\subsection{Embedding \& Vector Store}
\begin{itemize}
  \item \textbf{Embeddings} -- \textbf{Jina~AI} \texttt{jina-embeddings-v3} (default), accessed via the \texttt{LiteLLM} abstraction layer, generates dense vectors for each chunk.  The embedding model is configurable through environment variables; OpenAI \texttt{text-embedding-3-small} is also supported as an alternative provider.
  \item \textbf{Vector DB} -- ChromaDB (persisted to \texttt{./data/chromadb}) stores the embeddings with accompanying metadata (chunk index, title, source URL).  The vector index uses \textbf{cosine} similarity (\texttt{hnsw:space: cosine}).
\end{itemize}

\subsection{RAG Agent Design}
All agents share a common ``retriever $\to$ LLM'' pattern and differ in system prompts, temperature settings, and post-processing.  The active model is \texttt{groq/llama-3.3-70b-versatile} (Llama~3.3 70B hosted on Groq) via LiteLLM, but the system supports \textbf{multiple LLM providers}---OpenAI, Anthropic (Claude), and Groq---switchable through environment variables.  The output format throughout is a two-host dialogue between HOST\_A (main presenter) and HOST\_B (curious co-host).

\subsubsection{Script Generator Agent}
\begin{description}
  \item[Goal] Convert the ingested blog content directly into a natural, engaging two-host podcast dialogue (HOST\_A / HOST\_B) with an intro, sectioned body, and closing call-to-action.
  \item[Retrieval] Calls \texttt{retrieve\_all()} on the ChromaDB collection to provide full RAG context alongside the source text.  Source content is truncated to 8{,}000 characters and RAG context to 4{,}000 characters (first 10 chunks) to fit within LLM context windows.
  \item[Model] Llama~3.3 70B via Groq/LiteLLM (temperature 0.7, max\_tokens 4096).
\end{description}

\subsubsection{Accuracy Agent}
\begin{description}
  \item[Goal] Fact-check the generated script against the original source material.  Correct any misrepresentations, exaggerations, or hallucinated facts while preserving the dialogue format.
  \item[Retrieval] Retrieves source chunks from the same ChromaDB collection for comparison.  Source content is truncated to 8{,}000 characters.
  \item[Model] Llama~3.3 70B via Groq/LiteLLM (temperature 0.3, max\_tokens 4096) for high-precision verification.
\end{description}

\subsubsection{Storytelling Agent}
\begin{description}
  \item[Goal] Enhance narrative quality and pacing without changing facts.  Inserts prosodic and emotional cues: \texttt{[pause]}, \texttt{[emphasis]}, \texttt{[excited]}, \texttt{[thoughtful]}, \texttt{[serious]}, \texttt{[laughing]}.  Adds analogies, metaphors, and story-arc structure (beginning $\to$ tension $\to$ insight $\to$ resolution).
  \item[Model] Llama~3.3 70B via Groq/LiteLLM (temperature 0.6, max\_tokens 4096).
\end{description}

\subsubsection{Engagement Agent}
\begin{description}
  \item[Goal] Maximise listener engagement by adding rhetorical questions, direct listener addresses, section hooks, smooth transitions, and ``did you know'' moments.  Ensures a compelling opening hook and strong closing call-to-action.
  \item[Model] Llama~3.3 70B via Groq/LiteLLM (temperature 0.6, max\_tokens 4096).
\end{description}

All four agents are orchestrated via a \textbf{LangGraph} \texttt{StateGraph}.  A shared \texttt{GraphState} (\texttt{TypedDict}) carries the evolving script through the pipeline: \texttt{generate\_script} $\to$ \texttt{accuracy\_check} $\to$ \texttt{storytelling} $\to$ \texttt{engagement} $\to$ END.  A conditional edge after script generation aborts the pipeline if no script was produced; otherwise the remaining steps proceed linearly, with each node falling through gracefully on error.

\subsection{Voice Synthesis}
The system uses \textbf{Google Cloud Text-to-Speech} \cite{GoogleCloudTTS2024} for speech synthesis.  Each host is assigned a \textbf{Standard} voice: \texttt{en-US-Standard-J} (male) for ``Mike'' (HOST\_A) and \texttt{en-US-Standard-H} (female) for ``Sarah'' (HOST\_B).  Voice names and language codes are configurable via environment variables.

Before synthesis, a \texttt{dialogue\_parser} splits the final script into per-speaker segments, extracting cue markers (\texttt{[pause]}, \texttt{[emphasis]}, \texttt{[excited]}, etc.) as metadata.  The parser supports both the canonical \texttt{HOST\_A:}/\texttt{HOST\_B:} format (via \texttt{parse\_script()}) and natural speaker names such as ``Mike:'' or ``Sarah:'' (via \texttt{parse\_dialogue()}, which is used by the API routes).  Segments exceeding 500 characters are automatically split at sentence boundaries before synthesis.

During synthesis, cue markers are converted to \textbf{SSML} (Speech Synthesis Markup Language) tags: \texttt{[pause]} maps to \texttt{<break time="500ms"/>}, \texttt{[emphasis]} to \texttt{<emphasis level="strong">}, \texttt{[excited]} to \texttt{<prosody rate="fast" pitch="+2st">}, \texttt{[thoughtful]} to \texttt{<prosody rate="slow">}, and \texttt{[serious]} to \texttt{<prosody rate="slow" pitch="-1st">}.  Each segment is synthesised individually via the Google Cloud TTS API (\texttt{LINEAR16} encoding) and saved as a WAV file.  Synthesis runs are isolated in UUID-keyed subdirectories, enabling safe concurrent usage.

\textbf{Audio post-processing} (via \texttt{pydub}) applies volume normalisation, silence trimming ($-40$~dBFS threshold), and optional noise reduction (\texttt{noisereduce}).  The \texttt{assembler} merges all segments into a final podcast file, inserting 400\,ms pauses between different speakers, 800\,ms pauses for \texttt{[pause]} cues, and optional intro/outro music.

Output audio is exported as MP3 (192\,kbps) or WAV and stored on the local filesystem (\texttt{./output/audio/}).

\subsection{API Layer}
A lightweight FastAPI service (\texttt{api/main.py}) exposes endpoints under the \texttt{/api/v1/} prefix:

\begin{description}
  \item[\texttt{POST /api/v1/upload-blog}] Accepts a blog URL, raw text, or markdown; triggers ingestion and ChromaDB indexing; returns a job ID.
  \item[\texttt{POST /api/v1/generate-script}] Generates the two-host podcast script for a previously ingested blog (by job ID).
  \item[\texttt{POST /api/v1/generate-audio}] Synthesises audio from the generated script.
  \item[\texttt{POST /api/v1/generate-podcast}] Full pipeline (ingest $\to$ script $\to$ audio) in a single call, executed as a background task.
  \item[\texttt{GET /api/v1/job/\{job\_id\}}] Returns job status and progress.
  \item[\texttt{GET /api/v1/job/\{job\_id\}/script}] Retrieves the generated podcast script for the given job.
  \item[\texttt{GET /api/v1/download/\{job\_id\}}] Streams the generated audio file.
  \item[\texttt{GET /health}] Health-check endpoint at the root level.
\end{description}

Job state is currently tracked in an in-memory Python dictionary (with Redis or a database recommended for production).  The application is run directly via \texttt{uvicorn}.  CORS middleware is configured with permissive origins (\texttt{allow\_origins=["*"]}) for development, and a \texttt{/static} mount serves generated audio files.  At startup, stale TTS output directories older than one hour are automatically cleaned up.

\section{Implementation Details}
\label{sec:implementation}

This section describes the concrete technical choices, code organisation, and deployment workflow that realise the architecture presented in Section~\ref{sec:method}.

\subsection{Backend Framework}
The server-side application is built with \textbf{FastAPI} \cite{FastAPI2024}, a modern asynchronous Python web framework that provides automatic OpenAPI documentation, request validation via Pydantic models \cite{Pydantic2024}, and high throughput through Starlette's ASGI core.  The application is served by \textbf{Uvicorn} with live-reload support during development.

At startup a \emph{lifespan} handler ensures that output directories exist, logs the active configuration (LLM model, embedding model, TTS model), and cleans up any stale TTS synthesis directories older than one hour.  CORS middleware is configured with permissive origins (\texttt{allow\_origins=["*"]}) for development; production deployments should restrict this to known front-end origins.  A \texttt{/static} mount exposes the \texttt{./output} directory so that generated audio files can be served directly over HTTP.

\subsection{Frontend}
The user-facing interface is a \textbf{Streamlit} \cite{Streamlit2024} single-page application (\texttt{frontend/app.py}, $\approx$420 lines).  It provides:
\begin{itemize}[nosep]
  \item A three-tab blog input form accepting a \textbf{URL}, \textbf{raw text}, or \textbf{Markdown}.
  \item An optional episode-title field and a toggle between \emph{full-pipeline} (one-click) and \emph{step-by-step} modes.
  \item A real-time progress tracker that polls the backend every three seconds, displaying pipeline stage indicators (\checkmark, \textbullet, $\circ$) for each stage: \textit{ingesting}, \textit{generating\_script}, \textit{enhancing\_script}, \textit{synthesizing}, \textit{post\_processing}, and \textit{completed}.
  \item A script preview panel that renders HOST\_A and HOST\_B turns as chat-message bubbles with microphone and headphone icons.
  \item A built-in audio player and download buttons for both the generated script (\texttt{.txt}) and the podcast file (\texttt{.mp3}).
\end{itemize}
Communication with the backend is handled by a thin HTTP client wrapper (\texttt{frontend/api.py}) using the \texttt{requests} library with 120\,s POST and 30\,s GET timeouts.

\subsection{Database}
The system uses \textbf{ChromaDB} \cite{ChromaDB2024} as its sole persistent data store.  ChromaDB is configured as a \texttt{PersistentClient} backed by a local SQLite file (\texttt{./data/chromadb/chroma.sqlite3}).  Collections are created per blog using the naming convention \texttt{blog\_\{content\_hash\}} (where \texttt{content\_hash} is the first 16 hex characters of the SHA-256 digest of the blog text).  The HNSW index uses \textbf{cosine} similarity (\texttt{hnsw:space: cosine}).

Job state---tracking each pipeline run's status, progress percentage, and output paths---is maintained in an \textbf{in-memory Python dictionary}.  This is adequate for single-server development but should be replaced with Redis or a relational database (e.g., PostgreSQL) for production deployments where server restarts must not lose state.

\subsection{LLM Integration}
All LLM calls are routed through \textbf{LiteLLM} \cite{LiteLLM2024}, a unified abstraction layer that exposes a single \texttt{completion()} / \texttt{embedding()} interface across providers.  The system supports three LLM providers---\textbf{OpenAI}, \textbf{Anthropic (Claude)}, and \textbf{Groq}---switchable via the \texttt{LLM\_MODEL} environment variable (current deployment: \texttt{groq/llama-3.3-70b-versatile}).  Provider-specific API keys are routed automatically: setting \texttt{LLM\_API\_KEY} suffices; the code detects the provider prefix and sets the corresponding environment variable (\texttt{OPENAI\_API\_KEY}, \texttt{ANTHROPIC\_API\_KEY}, or \texttt{GROQ\_API\_KEY}).

Embedding generation similarly supports multiple providers through LiteLLM.  The current deployment uses \textbf{Jina~AI} \texttt{jina-embeddings-v3}; OpenAI \texttt{text-embedding-3-small} is also supported as an alternative.

Configuration is centralised in a \textbf{Pydantic} \texttt{Settings} class (\texttt{config/settings.py}) that loads all parameters from environment variables or a \texttt{.env} file, covering the LLM model, API keys, embedding model, ChromaDB path, Google Cloud TTS voice names, language code, output directories, audio pause durations, API host/port, and log level.

\subsection{Deployment Setup}
The system is designed for \textbf{local single-machine deployment}:
\begin{enumerate}[nosep]
  \item Install dependencies: \texttt{pip install -e .} (Python $\ge$ 3.11, build system: \texttt{hatchling}).
  \item Start backend: \texttt{uvicorn api.main:app {-}{-}reload} (binds to \texttt{0.0.0.0:8000}).
  \item Start frontend: \texttt{streamlit run frontend/app.py} (connects to \texttt{localhost:8000}).
\end{enumerate}
The Google Cloud TTS client is initialised lazily on the first synthesis request.  Each TTS run is isolated in a UUID-keyed subdirectory under \texttt{./output/audio/segments/}, enabling safe concurrent usage and automatic cleanup.

Cloud and containerised deployments (Docker Compose with separate API, worker, and ChromaDB services) are planned as future work (Section~\ref{sec:future}).

\subsection{Quality Evaluation}
Formal quality metrics (BLEU, ROUGE, MOS, PESQ) are defined in the Evaluation Plan (Table~\ref{tab:evaluation}) but have not yet been benchmarked on a curated corpus.  Below we provide a qualitative analysis based on manual inspection of generated scripts and audio.

\subsubsection{Accuracy}
The \textbf{Accuracy Agent} operates at temperature~0.3 and cross-references every claim in the generated script against the full set of source chunks (up to 8\,000 characters).  In manual spot-checks across five test blogs, no factual inaccuracies or hallucinated statistics were observed in the final scripts.  The RAG retrieval step ensures that only content grounded in the original blog is available to the generation prompt, providing a first line of defence against hallucination \cite{Lewis2020RAG}.

\subsubsection{Coherence}
Scripts exhibit a clear narrative arc---introduction, sectioned exploration of key topics, and a closing call-to-action.  The \textbf{Storytelling Agent} enriches this structure with prosodic cues (\texttt{[pause]}, \texttt{[emphasis]}, \texttt{[excited]}, \texttt{[thoughtful]}, \texttt{[serious]}, \texttt{[laughing]}) and rhetorical devices (analogies, metaphors).  The two-host format (HOST\_A as main presenter, HOST\_B as curious co-host) creates a natural question-and-answer rhythm.

\subsubsection{Relevance}
The Script Generator receives the top-10 retrieved chunks (4\,000 characters) alongside the truncated blog content (8\,000 characters), ensuring comprehensive topic coverage.  Manual review confirms that key themes, statistics, and insights from the source blogs are faithfully represented in generated scripts.

\subsubsection{Hallucination Mitigation}
The system employs a \textbf{three-layer defence}:
\begin{enumerate}[nosep]
  \item \textbf{RAG grounding} -- The LLM prompt includes only retrieved source chunks, not open-ended queries \cite{Gao2022RAGSurvey}.
  \item \textbf{Dedicated Accuracy Agent} -- A separate agent explicitly fact-checks the script against the source material at low temperature (0.3).
  \item \textbf{Conservative generation settings} -- The Accuracy Agent uses the lowest temperature in the pipeline, favouring precision over creativity.
\end{enumerate}
While this approach substantially reduces hallucination risk, it does not eliminate it entirely.  Formal hallucination-rate measurement on a labelled corpus remains as future work.

\section{Results and Discussion}
\label{sec:results}

This section provides an analytical assessment of system behaviour, moving beyond feature description to examine what the pipeline achieves, where it falls short, and how observed outcomes relate to the objectives stated in Section~\ref{sec:obj}.

\subsection{What Worked Well}

\paragraph{Multi-agent refinement.}
The sequential four-agent pipeline produces noticeably more natural and engaging scripts than a single monolithic prompt.  The raw script generated by the Script Generator is factually grounded but often reads as a flat summary.  Passing it through the Storytelling Agent adds narrative arc, prosodic cues, and rhetorical devices, while the Engagement Agent inserts listener-directed hooks, transitions, and ``did-you-know'' moments.  Comparing raw versus final scripts, the refinement stages typically add 20--30\% more speaking turns and substantially improve conversational flow.

\paragraph{RAG grounding.}
By constraining the LLM to retrieved chunks stored in ChromaDB, the system effectively limits the scope for hallucination.  Across manual inspections of five test blogs spanning technology, education, and health topics, the final scripts faithfully reflected the source material without introducing fabricated claims.

\paragraph{Voice consistency.}
Google Cloud TTS produces two clearly distinguishable host voices (``Mike'' via \texttt{en-US-Standard-J} and ``Sarah'' via \texttt{en-US-Standard-H}) that remain consistent across all segments of a given podcast episode.  Because voice identity is determined by the selected voice model rather than a reference recording, consistency is inherently guaranteed without additional engineering effort.

\paragraph{Modular architecture.}
Strict separation between ingestion, RAG, agents, TTS, and audio modules allowed each component to be developed, tested, and debugged independently.  The 42-test suite runs in under 10 seconds with fully mocked externals, providing rapid feedback during development.

\subsection{Problems Encountered}

\paragraph{API-based TTS latency.}
Google Cloud TTS API calls typically complete in 1--3 seconds per segment, resulting in 15--30 seconds for a full 10--15 segment podcast.  Unlike local model inference, there is no cold-start model loading delay.  The lightweight \texttt{TextToSpeechClient} initialises in under one second on the first request.

\paragraph{Content truncation for long articles.}
The Script Generator truncates blog content to 8\,000 characters and uses only the top-10 retrieved chunks (4\,000 characters).  For blog posts exceeding $\approx$5\,000 words, tail-end content may be under-represented in the generated script.  While the RAG retriever partially compensates by surfacing the most semantically relevant chunks regardless of position, important concluding sections can still be missed.

\paragraph{SSML limitations.}
Google Cloud TTS Standard voices support a subset of SSML tags.  While \texttt{<break>}, \texttt{<emphasis>}, and \texttt{<prosody>} are well supported, more nuanced emotional cues (e.g., laughter, whispering) cannot be directly expressed via SSML and are approximated with pitch/rate adjustments.

\paragraph{Ephemeral job state.}
The in-memory job dictionary loses all pipeline state on server restart.  Any in-progress jobs are silently abandoned, and completed jobs become irretrievable.  For a prototype this is acceptable, but production use requires persistent job storage.

\subsection{Limitations}

\begin{itemize}[nosep]
  \item \textbf{English-only.}  Although Google Cloud TTS supports dozens of languages, the agent prompts, dialogue parsing logic, and sentence segmentation are all English-specific.  Multilingual support would require localised prompts and parser adjustments.
  \item \textbf{No streaming output.}  Users must wait for the full pipeline to complete before accessing any output.  Progressive delivery---streaming script chunks as they are generated, or streaming audio segment-by-segment---is not yet implemented.
  \item \textbf{No script editing.}  Generated scripts cannot be manually edited and re-synthesised through the UI.  Users who wish to correct a factual error or adjust tone must re-run the entire pipeline.
  \item \textbf{Single-machine deployment.}  The system runs on a single server with no horizontal scaling.  Concurrent users share the same API rate limits, which may require quota management for high-traffic deployments.
  \item \textbf{No source verification.}  The system trusts the input blog content.  If the source material itself contains misinformation, the pipeline will faithfully reproduce it as a podcast.
\end{itemize}

\subsection{Observed Model Behaviour}

\begin{itemize}[nosep]
  \item \textbf{Llama~3.3 70B via Groq (temperature 0.7)} produces diverse and readable scripts but can be verbose.  The Engagement Agent's 120\% length cap prevents unbounded script inflation.
  \item \textbf{Accuracy Agent (temperature 0.3)} is appropriately conservative---it corrects factual deviations but sometimes overcorrects stylistic choices (e.g., replacing a vivid metaphor with a plain statement), slightly flattening the narrative tone.
  \item \textbf{Sequential agent execution} means each agent operates on the full output of the previous, creating a compounding refinement effect.  However, errors introduced by one agent propagate downstream unless caught by the next.
  \item \textbf{Google Cloud TTS (Standard voices)} produces clear, intelligible speech.  SSML tags provide useful prosody control, though Standard voices are less expressive than WaveNet or Neural2 alternatives.  Upgrading to higher-tier voices is a straightforward configuration change.
\end{itemize}

\section{Ethical Considerations}
\label{sec:ethics}

Deploying AI-generated podcast content raises several ethical concerns that practitioners and users must be aware of.

\subsection{Bias Risks}
Large language models reflect biases present in their training corpora \cite{Bender2021Parrots}.  When a blog article discusses sensitive topics---politics, health, social issues---the LLM may amplify or introduce subtle biases during script generation.  The Accuracy Agent mitigates this partially by cross-referencing the source material, but it cannot detect biases already present in the source blog itself.  Future work could integrate a dedicated bias-detection layer that flags potentially skewed language before synthesis.

\subsection{Hallucination Risks}
Despite the three-layer defence described in Section~\ref{sec:testing} (RAG grounding, Accuracy Agent, low temperature), LLMs can still fabricate plausible-sounding but incorrect details, particularly when the source content is ambiguous, sparse, or self-contradictory \cite{Weidinger2022Taxonomy}.  Users should treat AI-generated podcasts as supplementary content rather than authoritative sources and verify critical claims against the original blog.

\subsection{Data Privacy}
The system processes user-supplied blog content in two ways that raise privacy considerations:
\begin{enumerate}[nosep]
  \item \textbf{Third-party API transmission.}  Blog text and generated scripts are sent to external LLM providers (OpenAI, Anthropic, or Groq) for processing.  Users should be informed that their content passes through third-party infrastructure subject to each provider's data-handling and retention policies.
  \item \textbf{Local storage.}  ChromaDB persists chunked blog content and embeddings on the local filesystem.  While no personally identifiable information (PII) is explicitly collected, blog articles may contain PII or proprietary content that persists in the vector store until manually deleted.
\end{enumerate}
For sensitive content, the LiteLLM abstraction supports self-hosted LLM deployments (e.g., via Ollama), which keep all data on-premises.

\subsection{Responsible AI Usage}
\paragraph{Transparency.}  Generated podcasts should be clearly labelled as AI-generated content.  Listeners may not be able to distinguish synthesised speech from human recordings, making disclosure essential to maintain trust.

\paragraph{Voice synthesis ethics.}  Google Cloud TTS uses pre-built voice models provided by Google, eliminating voice-cloning risks associated with reference-audio-based systems.  Because the system does not clone real human voices, there is no risk of unauthorised impersonation via the TTS component.  However, AI-synthesised speech can still be mistaken for human speech, so clear labelling of AI-generated content remains important.

\paragraph{Content accountability.}  The system cannot verify the truthfulness of the source blog.  If the input material contains misinformation, propaganda, or harmful content, the pipeline will faithfully convert it into an audio format, potentially broadening its reach.  Operators should implement content moderation or input filtering for public-facing deployments.

\subsection{Limitations of Generative Systems}
Output quality is fundamentally bounded by input quality: poorly structured or factually dubious blogs produce lower-quality podcasts.  The system cannot perform original research, critical analysis, or fact-checking beyond the source material.  Furthermore, the conversational format may create an illusion of authority (two ``experts'' discussing a topic) that the original blog post did not claim.

\section{Future Work}
\label{sec:future}

Several avenues for improvement and extension are planned.

\subsection{Quantitative Evaluation}
Implementing the evaluation framework defined in Table~\ref{tab:evaluation} is the most immediate priority.  This involves curating a benchmark corpus of 50+ blog articles with human-written podcast scripts, measuring content fidelity via BLEU and ROUGE, assessing audio clarity via PESQ, and conducting Mean Opinion Score (MOS) listening tests with human evaluators.

\subsection{Scalability}
\begin{itemize}[nosep]
  \item Replace the in-memory job store with \textbf{Redis} or \textbf{PostgreSQL} for persistent, crash-resilient state management.
  \item Introduce a \textbf{Celery} or \textbf{RQ} task queue to decouple pipeline execution from the web server, enabling horizontal scaling of worker processes.
  \item Containerise the application with \textbf{Docker Compose}, packaging the API server, task workers, ChromaDB, and Redis as separate services for cloud deployment.
\end{itemize}

\subsection{Model Fine-Tuning}
\begin{itemize}[nosep]
  \item \textbf{LLM fine-tuning.}  Train a smaller open-source model (e.g., Llama~3, Mistral) on a dataset of human-edited podcast scripts to reduce dependence on commercial APIs and lower per-conversion latency and cost.
  \item \textbf{TTS voice upgrade.}  Upgrade from Google Cloud TTS Standard voices to WaveNet or Neural2 voices for improved prosody, intonation, and naturalness at a modest cost increase.
\end{itemize}

\subsection{Additional Integrations}
\begin{itemize}[nosep]
  \item \textbf{RSS feed ingestion} for automated, recurring podcast series from blog feeds.
  \item \textbf{YouTube transcript support} to convert video content into podcast format.
  \item \textbf{Multilingual support}, leveraging Google Cloud TTS's multi-language voice catalogue together with multilingual LLMs and localised prompt templates.
  \item \textbf{Webhook notifications} to alert users or downstream systems upon pipeline completion.
\end{itemize}

\subsection{Performance Optimisation}
\begin{itemize}[nosep]
  \item \textbf{Parallel agent execution.}  The Storytelling and Engagement agents operate on independent quality dimensions and could potentially execute concurrently, reducing pipeline latency by 25--30\%.
  \item \textbf{Streaming TTS synthesis.}  Begin audio playback of early segments while later segments are still being synthesised, reducing perceived wait time.
  \item \textbf{Parallel TTS requests.}  Synthesise multiple segments concurrently via the Google Cloud TTS API to reduce total synthesis time for long scripts.
  \item \textbf{Embedding cache.}  Cache embeddings for previously ingested content to skip re-computation on repeat conversions of the same blog.
\end{itemize}

\subsection{User Experience Improvements}
\begin{itemize}[nosep]
  \item An in-browser \textbf{script editor} allowing users to revise the generated dialogue before synthesis.
  \item \textbf{Voice selection}, enabling users to choose from Google Cloud TTS's extensive voice catalogue or switch between Standard, WaveNet, and Neural2 voice tiers.
  \item \textbf{Episode metadata generation} including show notes, chapter timestamps, and SEO-friendly descriptions.
  \item \textbf{Batch processing} for converting multiple blog articles into a podcast series in a single operation.
\end{itemize}

\section{Conclusion}
\label{sec:conclusion}

This work addressed the labour-intensive problem of converting written blog content into podcast-ready audio by designing and implementing a full-stack, RAG-based multi-agent pipeline.  The system ingests blog articles from URLs, raw text, or Markdown; builds a ChromaDB vector store of semantically meaningful chunks; orchestrates four specialised agents---Script Generator, Accuracy Agent, Storytelling Agent, and Engagement Agent---through a LangGraph state machine; and synthesises expressive two-voice speech using Google Cloud Text-to-Speech with Standard voices and SSML prosody control.

Key findings from development and testing include: (1)~RAG grounding effectively constrains LLM output to source material, substantially reducing hallucination risk; (2)~multi-agent sequential refinement produces measurably more natural, engaging, and structurally sound scripts than single-prompt generation; and (3)~Google Cloud TTS with SSML cue mapping achieves consistent, distinguishable dual-host audio with low-latency API-based synthesis.

The modular architecture---with a FastAPI backend, Streamlit frontend, and provider-agnostic LLM/embedding integration via LiteLLM---provides a foundation for future extensions including multilingual support, scalable cloud deployment, and model fine-tuning.  While formal quantitative benchmarks (BLEU, ROUGE, MOS, PESQ) remain as planned evaluation tasks, the system demonstrates the viability of an end-to-end, open-source blog-to-podcast pipeline that bridges written and audio content, with practical applications in accessibility, education, and content repurposing.
